# 项目代理协作规则

本文件适用于 `E:\手机收银软件开发` 整个工作区。任何主代理或子代理开始任务、恢复任务或在上下文压缩后继续工作时，都必须先遵守本文件。

## 开始与恢复时的必读顺序

1. 先读 `docs/ACTIVE_ITERATION.md`，确认唯一的当前阶段、任务编号、暂停点、下一动作和阻塞项。
2. 再读 `修改方案/ms2011_live_product_promotion_sync_implementation_plan.md` 中当前阶段和任务编号的完整章节，包括前置条件、允许文件、禁止范围、完成标准和依赖顺序。
3. 涉及产品行为或安全边界时，读 `修改方案/ms2011_live_product_promotion_sync_plan.md`。
4. 只在需要历史证据时读 `docs/PROJECT_STATUS.md`、`docs/PROJECT_LOG.md` 和 `修改方案/实施证据`、`修改方案/促销验证` 下的对应文件。
5. 不得仅凭旧会话摘要、旧状态标题或单个日志片段判断当前阶段。

## 权威顺序

发生冲突时，按以下顺序处理：

1. 用户在当前任务中的明确指令和批准边界。
2. 产品设计基线中的产品范围与安全边界。
3. 细化实施计划中的合同、门禁、任务依赖和文件所有权。
4. `docs/ACTIVE_ITERATION.md` 中的当前动作和暂停点；它不能覆盖上面两份计划。
5. `docs/IMPLEMENTATION_STATUS.md`、`docs/PROJECT_STATUS.md`、`docs/PROJECT_LOG.md` 和历史证据。

实施计划中若同时存在旧顶部摘要和较新的带日期完成/暂停记录，以同一文件内较新的阶段记录为准，并由主协调器尽快修正顶部摘要。

## 活动检查点纪律

- `docs/ACTIVE_ITERATION.md` 是唯一的短期动作状态文件，采用覆盖式维护，不作为无限追加日志。
- 只有主协调器可以常规更新该文件。子代理只返回结果；除非任务明确授权，不直接修改活动检查点。
- 主协调器必须在以下时点更新它：开始一个交付批次或阶段前、出现影响恢复点的任务结论时、开始长时间构建或外部操作前后、出现阻塞时、请求用户批准前、阶段完成、暂停或交接前。同一阶段内无阻塞且文件边界不变的连续小任务不要求为每个微步骤重复改写活动文件，但必须始终保留可恢复的当前任务、已验收任务和下一精确动作。
- 每次更新至少记录：阶段和任务编号、状态、已完成检查点、当前部分修改、最后验证结果、下一精确动作、阻塞/待批准事项、验收条件和回退点。
- 每个任务编号完成后，主协调器必须核对实际 diff、定向测试、边界和回退证据，并在活动检查点中保留紧凑结论。若同一阶段的后续任务已由实施计划明确解锁，可以标记为 `TASK_PASS_PENDING_STAGE_SYNC` 后继续；这不等于阶段完成或发布通过。
- 每个 S 阶段结束时，主协调器必须在活动检查点写明该阶段的任务、定向/阶段测试、风险、回退点和门禁结论。若阶段属于跨阶段交付批次且产品门禁已通过，可以标记 `STAGE_PASS_PENDING_BATCH_SYNC` 后进入同一批次的下一阶段；每个交付批次结束时再一次性把该批次全部结果同步到实施计划当前状态/带日期记录、`docs/IMPLEMENTATION_STATUS.md`、`docs/PROJECT_STATUS.md`、`docs/PROJECT_LOG.md` 和其他受影响的方案索引，最后更新活动文件。批次文档未同步时只能标记“待批次文档同步”。
- 恢复工作时不得跳过 `Next exact action`；若现场代码与活动文件不符，先停止并报告差异。

## 精简执行模式（2026-07-18 用户批准）

- 精简只改变调度、复核、测试、构建、hash 和文档同步频率；不得删除任务编号、产品范围、任务依赖、文件所有权、安全边界、真实电脑/真机验收或最终完成标准。
- 每次开始执行一个新任务或交付批次前，主协调器必须先向用户报告本次实际调用的子代理数量、各自任务、模型和推理强度；不调用子代理时也必须明确报告 0 个。
- 主协调器默认采用最小充分团队：普通任务最多一个写代理；只有合同存在实质歧义时增加一个前置只读审查代理；独立后置复核集中在高风险任务和阶段门禁，不为相同证据重复安排多个审查代理。
- 必须在稳定阶段/工作包快照上独立复核的高风险节点至少包括：MB-04；S09 的 MB-05/MB-06；S10 的 MB-07、CB-01～CB-03 订单边界集群；S11 的 PM-01～PM-04 和手动定价审计；每个准备上线促销类型的 PV/PN/PE 工作包、PV/PN/PE-ORDER、PN-INTEGRATION、PE-REGISTRY 和全部 PI；IV-02、IV-04；S17/G9。一次独立复核会话可以覆盖同一稳定快照上的多个串行任务或互不重叠工作包，但必须逐任务/工作包给出 verdict；不得为同一快照重复启动等价审查。其他任务由主协调器检查实际文件、diff 和定向测试，发现跨端合同、文件能力、订单价格或促销安全风险时升级独立复核。
- 若任务合同或 `ACTIVE_ITERATION` 已明确要求不同职责的多个独立 verdict，精简模式不得合并或删除；当前 MB-04 必须分别取得独立安全审查和独立功能/制品验证 PASS。
- 测试采用三级节奏：每个任务运行新增/变更测试、直接相关模块测试及必要编译；每个 S 阶段运行该端完整回归和编译；S10、S15、S16、S17 运行跨端或全链路验证。涉及 I/O、安全边界或崩溃恢复的任务仍必须覆盖失败、取消、损坏或恢复路径。
- Android APK 和阶段制品只在实施计划指定的阶段门禁构建一次；PC EXE/ZIP 只在 G9/S17 正式构建。代码或测试在构建后发生变化时，原构建证据立即失效，必须重建。
- 每个任务只计算本轮修改文件和明确冻结边界文件的 SHA-256；阶段门禁计算阶段源码/fixture 清单；S17 计算全部发布制品、内容清单和发布副本 hash。
- 子代理的口头结论不能代替主协调器核对。精简模式减少重复执行，不降低失败关闭、独立证据或最终验收强度。

## 子代理调度与文档可读性

- 整个任务累计创建、复用或轮换的子代理数量不设上限；项目目标并发槽位为 10。主协调器根据任务复杂度、依赖和文件所有权自行决定每轮实际使用数量，不要求填满槽位，也不为节省代理数量牺牲独立复核。
- 只把真正独立、边界明确且并行收益高的工作交给不同代理；不得把同一合同或同一稳定快照重复交给多个代理做等价审查。拆分必须保留明确的前置条件、允许文件、禁止范围、验收命令和返回证据格式。
- 同一时刻的实际并发仍必须服从当前会话已经加载的物理槽位和工具限制。能够独立并行的任务才并行；存在文件、合同、门禁或状态依赖时仍必须串行，不得为了提高并发跳过门禁。
- 主协调器保留总体设计、任务拆分、冲突处理、最终验收和状态同步责任；子代理不得自行进入下一任务、修改全局状态文档或绕过门禁。
- 每个实现代理一次仍只领取一个明确任务编号；同一文件或紧耦合模块不得由多个写代理并行修改。
- 每个任务编号完成后必须先形成可恢复的紧凑证据；同一阶段的任务在阶段结束时集中写入 `PROJECT_LOG`。`PROJECT_STATUS` 只在能力、门禁或发布物变化时更新；实施计划与方案索引只在阶段/交付批次边界更新，避免复制整段日志。
- 批次文档同步属于交付批次完成条件。单个任务代码和测试通过后可标记 `TASK_PASS_PENDING_STAGE_SYNC`；S 阶段产品门禁和阶段证据通过后可标记 `STAGE_PASS_PENDING_BATCH_SYNC`。仅在依赖表明确允许时继续同阶段下游任务；仅在精简执行表明确属于同一交付批次时跨到下一 S 阶段。不得跨越未通过的阶段产品门禁、未关闭的交付批次或宏观门禁。

## 任务与文件边界

- 每个实现代理一次只执行一个明确任务编号，不得把整个 S 阶段一次领取。
- 修改前确认实施计划列出的前置条件和允许文件。未列出的模块默认禁止修改。
- 多个写代理不得同时编辑同一文件或紧耦合模块。
- `android-emergency-pos` 和 `pc-sync-tool` 是实施目录；`mobile_pos_publish` 是发布副本，除发布任务外不作为开发主目录。
- 工作区根目录、Android 和 PC 实施目录没有有效 Git 历史。不得假定 `git status` 能识别用户修改；活动检查点必须列出本轮触碰文件，必要时记录 SHA-256。
- 不覆盖、清理或重置不能确认归属的既有文件、缓存、产物或用户修改。

## MS2011 硬安全边界

- G0B 未通过时，正式应用不得连接真实 MS2011、自动实时读取或生产发布。
- SQL 只允许实施计划批准的固定 QueryId `SELECT`；禁止任意 SQL 输入和所有写入、DDL、维护、权限修改或写失败测试。
- 禁止复制、移动、改名或删除运行中的 MDF/LDF，禁止控制 SQL Server 或鸣盛进程，禁止写入 EPSA/Espsa/鸣盛目录。
- 禁止程序自动修改防火墙规则，禁止手机直连 SQL Server。
- 不猜测促销公式、优先级、叠加、余数、日期/时段边界或舍入。只有 `VERIFIED` 规则可以自动计算。
- 当前目标身份结论为 `WRITE_CAPABILITY_PRESENT`，不能因为 ODBC 接受 read-only 提示而视为安全身份。

## 验证与报告

- “通过”必须附实际命令、通过数量和本轮结果；历史测试记录不能冒充当前执行。
- 需要写构建输出时，由拥有相应写权限且获得授权的代理执行。
- 每个任务报告必须包含：修改文件、测试/构建证据、安全边界、遗留风险、回退点和下游是否解锁。
- 普通任务报告定向测试与必要编译；完整回归、制品构建和全量 hash 以最近一次代码未再变化的阶段门禁结果为准，不得在每个小任务中机械重复，也不得用代码变化前的结果充当当前证据。
- 未满足全部阶段条件时只能标记进行中、部分完成或阻塞，不得提前进入下一阶段。
