# MS2011 实时商品与促销只读同步修改方案

更新时间：2026-07-13  
适用项目：`pc-sync-tool` + `android-emergency-pos`  
任务类型：电脑端后端、电脑端前端、手机端后端、手机端前端、促销黑盒验证、打包与端到端验收  
目标读者：开发人员、验收人员、后续维护人员

## 1. 文档状态与本轮目标

本文是经过逐项确认后的设计基线。后续实施、验收和项目文档更新均应以本文为准；如与早期方案或分析结论冲突，以本文为准。

本轮目标是在不修改、不停止、不妨碍鸣盛电脑收银系统的前提下，实现：

1. 电脑端同步工具从实时运行的 SQL Server 2000 数据库 `MS2011` 中只读提取商品、分类、单位，以及已经配置、能够关联商品并可能参与鸣盛实际收银定价的促销候选数据。
2. 电脑端把读取结果规范化为独立的 SQLite v2 快照，经完整性验证和两次一致性校验后原子发布。
3. 手机端在前台按独立周期检查新快照，后台下载和验证，并只在订单边界切换快照。
4. 手机端离线时继续使用最后一份有效快照。
5. 手机端按照阿根廷时间计算已验证促销；结果必须与鸣盛电脑端黑盒测试一致。
6. 未验证、异常或无法安全解释的促销不得被静默忽略；相关商品自动结账受控，收银员可明确选择手动价格。
7. 保留现有 `AGT_MAIN.db` 备份库同步方式，作为独立兼容模式。

这里的“实时同步”是安全的准实时同步，不是 SQL 写入后的即时推送。默认设置且数据库空闲、网络正常时，电脑端修改通常应在 60 秒内出现在手机的新购物车中；数据库繁忙时宁可延迟，也不能影响电脑端收银。

## 2. 已确认事实与方案变更

### 2.1 实时主库位置与数据来源

- `MS2011.MDF` / `MS2011.LDF` 是 SQL Server 2000 正在使用的实时主库文件。
- 正在运行时不能直接复制 MDF/LDF 作为同步方式。
- SQL 查询分析器已经确认数据库名为 `MS2011`。
- Windows 集成身份验证可用；此前 `bcp -T` 已成功只读导出数据，因此本方案不保存 SQL 用户名或密码。
- 当前导出的实时数据包含 11,168 个商品，晚于 EPSA 拷贝中的 11,141 个商品快照，证明它来自实时主库而不是每日备份。

### 2.2 已确认的数据表

商品、分类和单位：

```text
MS_GOODLIST
MS_GOODTYPELIST
MS_UNITLIST
```

复杂促销相关表：

```text
MS_CUXIAO_GOOD
MS_SALE_CXDAN1
MS_SALE_CXDETAIL1
MS_SALE_CXMASTERDING
MS_SALE_CXMASTERFOUR
MS_SALE_CXTABLE1
MS_SALE_CXTABLEDING
MS_SALE_CXTABLEFOUR
MS_SALE_WEEKDETAIL1
MS_SALE_WEEKDING
```

### 2.3 对旧分析结论的明确修订

`MS2011_商品分类单位促销数据分析.md` 曾按当时决定写明“手机端只使用简单促销，复杂促销不进入同步”。该范围已被后续决定替代。

本方案的新基线是：

- 同步鸣盛数据库中已经应用的促销候选及其日期、星期和时段规则；不把全部历史促销记录无条件发送到手机。
- 每种促销必须通过鸣盛电脑端黑盒测试确认语义、优先级、叠加关系和小数处理。
- 未验证规则保留原始数据但不得自动计算。
- 不再假定简单促销与复杂促销不会冲突。

## 3. 已确认产品决策

### 3.1 同步方向与数据权威

- 同步方向固定为电脑端 `MS2011` → 电脑同步工具 → 手机端。
- 电脑端是同步商品的唯一数据权威。
- 手机端不向 `MS2011` 写回任何商品、促销或交易数据。
- 电脑同步商品在手机端只读，并显示“由电脑端管理”。
- 手机本地创建的商品仍可编辑、删除；保留以后扩展手机编辑同步商品的入口，但本轮不实现写回。
- 当前使用一部手机；协议自然允许多部手机独立读取同一快照，不做设备间销售同步。

### 3.2 同步范围

本轮同步：

- 商品主键、条码、名称和必要搜索字段。
- 当前售价及促销计算需要的原始价格字段。
- 分类、单位。
- 停用状态、更新时间。
- 商品简单促销字段。
- 已经配置、能够关联商品且可能参与实际定价的复杂促销主记录、商品映射、数量阶梯、金额/折扣参数、日期、星期和时段。

本轮不同步：

- 库存数量。
- 会员、供应商、采购、销售流水、财务数据。
- 手机销售记录到电脑端。
- 鸣盛配置、授权、报表、打印模板。

### 3.3 金额、折扣与数量

- 商品价格和促销金额保持鸣盛原始十进制数值，电脑端使用 `Decimal`，手机业务模型使用 `BigDecimal`。
- `2099.99` 必须保存为 `2099.99`，不得转换成 `209999`。
- SQLite v2 使用规范化十进制文本保存业务金额和数量，不使用二进制 `REAL/double` 参与结账；不得改成“分”的整数模型。
- 折扣参数保留鸣盛表中的原始十进制数值，不转换成整数基点。
- 在黑盒测试确认前，不猜测 `5` 表示 `5%`、`95%`、`0.05` 或其他含义。
- 手机端允许手动输入小数数量，例如 `1.25`。
- 非整数数量自动排除全部促销，按 `数量 × 商品单价`计算。
- 整数数量只有在电脑端确实关联有效促销时才应用促销。
- 最终金额的小数位和舍入时点必须通过鸣盛黑盒结果确定，不得凭经验设定。

### 3.4 停用商品

- 停用商品记录保留在快照中，不物理删除。
- 正常搜索隐藏停用商品。
- 扫描停用商品条码时明确提示“商品已停用”。
- 停用商品不能加入新购物车。
- 已经存在于非空购物车中的商品继续使用该购物车创建时的旧快照，不在收银中途被移除或改价。

## 4. 硬性安全边界

这一节是强约束。任何实现和验收都必须遵守。

电脑端同步工具只允许：

- 使用 Windows 集成身份验证连接本机或已明确配置的 `MS2011` 实例。
- 执行程序内置白名单中的只读 `SELECT`。
- 使用只读系统元数据查询进行开发诊断或表结构兼容检查。
- 在同步工具自己的 AppData 目录中写入配置、日志、临时文件、历史快照、`latest.db` 和 manifest。
- 通过现有局域网 HTTP 服务向手机提供只读下载。

电脑端同步工具禁止：

- 禁止执行 `INSERT`、`UPDATE`、`DELETE`、`MERGE`、建表、删表、改表、存储过程执行和数据库配置命令。
- 禁止直接复制、移动、重命名、删除或替换正在运行的 `MS2011.MDF` / `MS2011.LDF`。
- 禁止分离、附加、备份、还原、收缩、修复或重新索引 `MS2011`。
- 禁止停止、暂停或重启 SQL Server 服务。
- 禁止停止、暂停、关闭或控制鸣盛收银软件。
- 禁止在 `EPSA`、`Espsa` 或鸣盛安装目录内创建同步文件。
- 禁止修改鸣盛软件、配置、授权、模板、报表、打印文件和原始数据库。
- 禁止为了同步成功而等待或持有会阻塞收银事务的锁。
- 禁止手机端通过任何同步接口写入电脑或 SQL Server。
- 禁止程序启动时静默修改 Windows 防火墙。

`READ UNCOMMITTED` / `NOLOCK` 不能从技术上保证“绝对零锁”，但本方案的验收标准是：同步查询不得阻塞鸣盛业务事务，不得造成可感知卡顿；出现繁忙、超时或不一致时立即放弃本轮并保留旧快照。

## 5. 总体架构

```text
实时 MS2011 SQL Server 2000
  → 只读连接和安全门
  → 变化探测与空闲窗口
  → 两次确定性读取
  → 规范化、关联校验、哈希比对
  → SQLite v2 临时快照
  → 完整性校验
  → 原子发布 latest.db + v2 manifest
  → 现有 Token HTTP 服务
  → 手机下载、SHA-256 和结构校验
  → 订单边界原子启用
  → 商品查询、收银和本地促销计算
```

复用现有能力：

- 电脑端托盘程序、配置、事件日志、AppData 路径、历史保留、发布锁、SHA-256、HTTP 服务和 Token。
- 手机端电脑连接配置、`/health`、下载、SHA-256、离线商品库和阿根廷时间工具。

新增能力：

- SQL Server 2000 只读适配器和查询白名单。
- 数据库繁忙检测、空闲窗口和双读一致性校验。
- MS2011 规范化器、促销关联验证器、SQLite v2 生成器。
- v2 manifest 和版本兼容判断。
- 手机端 v2 导入器、版本化商品/促销快照和订单边界切换器。
- 支持精确十进制金额与小数数量的价格模型。
- 经黑盒验证的多类型促销引擎。

## 6. 电脑端安全读取流程

### 6.1 自动检测周期

- 电脑端默认每 15 秒检测一次。
- 支持关闭自动检测。
- 支持预设值和自定义值，范围为 5 秒至 24 小时。
- 检测周期仅决定何时尝试，不代表数据库繁忙时必须执行。

### 6.2 变化探测与空闲窗口

推荐流程：

```text
1. 到达检测周期
2. 执行低成本只读变化探测
3. 未变化：结束本轮
4. 有变化：进入约 10 秒安静窗口
5. 安静窗口内仍有变化或数据库繁忙：结束本轮或继续等待下次周期
6. 数据库稳定：开始第一次完整读取
7. 规范化并计算 hash A
8. 再次检查数据库状态
9. 开始第二次完整读取
10. 规范化并计算 hash B
11. hash A != hash B：丢弃本轮
12. hash 相同且全部校验通过：生成并发布新快照
```

变化探测不能只依赖 `GUpdateTime`。实现应同时保留周期性全量指纹，以捕获未正确更新 `GUpdateTime` 的商品、分类、单位和促销变更。

### 6.3 查询约束

- 连接使用 Windows 集成身份验证。
- 自动提交，不开启长事务。
- 使用 `READ UNCOMMITTED` 或等价的 SQL Server 2000 低干扰读取方式。
- 所有查询设置短连接、短命令超时。
- 查询必须按稳定主键排序，确保两次规范化结果可比较。
- 只读取本方案列出的表和字段；表名、列名不得由外部输入任意拼接。
- 任何超时、连接错误、活动写入迹象、行数异常、关系异常或双读不一致都不得发布快照。

### 6.4 手动“立即同步”

- 点击后立即发起一次安全尝试。
- 数据库空闲则执行。
- 数据库繁忙则进入“等待空闲”队列。
- 用户可以取消等待。
- 不提供“强制同步”或“忽略安全检测”按钮。
- 手动同步与自动同步共用同一安全门和发布流程。

## 7. MS2011 数据映射

### 7.1 商品

稳定标识和主要字段：

```text
GID             跨端商品主键
GBarcode        扫码键
GFuBarCode      辅助条码原始值
GNameX          商品名称
GSalePrice      当前售价
GClass          分类代码
GUnit           单位代码
GStopFlag       停用状态
GUpdateTime     源更新时间
GHuiPrice       简单促销原始价格
GHuiPriceCount  简单促销原始门槛
```

具体同步列在实现前由字段白名单固定，未使用字段不得整行无选择地透传到手机。

### 7.2 分类和单位

关联方式固定为：

```text
MS_GOODLIST.GClass = MS_GOODTYPELIST.RTypeCode
MS_GOODLIST.GUnit  = MS_UNITLIST.UNumCode
```

- 分类和单位每次可以全量读取并参与快照哈希。
- 单位为空时，手机显示可以回退为“UN/件”，但快照仍保留原始空值。
- 不根据名称自动重分类或修改源值。

### 7.3 促销类型

当前数据已经识别出：

1. 商品简单数量促销：`GHuiPrice + GHuiPriceCount`。
2. 按数量折扣：`MS_SALE_CXDAN1 + MS_SALE_CXDETAIL1` 及关联映射和时段。
3. 指定数量固定总价：`MS_SALE_CXMASTERDING + MS_SALE_CXTABLEDING` 及关联映射和时段。
4. 多商品组合固定总价：`MS_SALE_CXMASTERFOUR + MS_SALE_CXTABLEFOUR` 及关联映射。

如果后续实时主库出现新的促销表值、规则形态或未见过的类型：

- 快照记录为未知/未验证类型和验证问题。
- 不猜测计算公式。
- 只阻止受该促销影响商品的自动结账，不影响其他商品。

## 8. SQLite v2 快照结构

新 `latest.db` 是电脑同步工具生成的规范化快照，不是原始鸣盛备份，也不伪装为 `AGT_MAIN.db`。

建议主表：

```text
sync_metadata
products
categories
units
promotions
promotion_products
promotion_tiers
promotion_schedules
validation_issues
```

### 8.1 关键设计规则

- 商品、分类、单位和促销使用稳定源主键。
- 商品价格、促销金额、折扣参数使用规范化十进制文本，保留原始数值并避免二进制浮点误差。
- 数量字段支持精确十进制；来自只允许整数的促销阶梯仍保留源字段语义。
- 每条促销保存类型、源表、源主键、名称、起止日期、原始参数、验证状态。
- 促销商品、阶梯和时段拆表保存，不把复杂规则塞进商品单行。
- `validation_issues` 记录不完整映射、缺失明细、未知类型和异常参数，不记录 Token 或敏感连接信息。
- 快照内保存 `schema_version`、`snapshot_id`、生成时间、源类型、源指纹和规范化哈希。
- 跨设备时间继续使用 UTC ISO 保存和传输；所有用户可见时间统一转换为阿根廷时间并标明 `ART`。

### 8.2 发布前校验

- 表结构和 `schema_version` 正确。
- 主键非空且唯一。
- 商品条码满足既定唯一性策略。
- 分类、单位关联可解释。
- 促销主表、商品映射、阶梯和时段引用完整，或明确进入 `validation_issues`。
- 记录数量和 manifest 汇总一致。
- 临时数据库可重新打开并执行完整性检查。
- SHA-256 计算完成。

只有全部硬性校验通过才能原子替换当前 v2 快照。任何失败均保留上一份有效快照。

## 9. HTTP v2 协议与旧模式兼容

保留：

```text
GET /health?token=...
```

新增建议端点：

```text
GET /v2/manifest.json?token=...
GET /v2/latest.db?token=...
```

v2 manifest 至少包含：

```text
schemaVersion
snapshotId
sourceType
createdAt
sizeBytes
sha256
minimumAppVersion
productCount
categoryCount
unitCount
promotionCount
validationIssueCount
downloadPath
```

兼容规则：

- 现有旧端点继续服务 `AGT_MAIN` 兼容模式。
- 电脑端可以选择 `AGT_MAIN 备份库` 或 `MS2011 实时主库`，同一时刻只有一个主同步模式。
- 手机仅启用自己支持的 `schemaVersion`。
- 不支持的版本给出明确升级提示，不尝试部分导入。
- Token 校验、局域网监听策略和不记录查询字符串的安全规则保持不变。

## 10. 手机端快照与订单边界

### 10.1 自动检查

- 手机端周期独立于电脑端。
- 默认 30 秒。
- 支持关闭、预设值和 5 秒至 24 小时自定义周期。
- 仅 App 位于前台时定时检测。
- App 恢复前台、进入收银页面、新建订单时立即检查一次。

### 10.2 下载与启用

```text
检查 v2 manifest
→ snapshotId 未变化：结束
→ 后台下载临时文件
→ 校验大小、SHA-256、schemaVersion、结构和记录汇总
→ 校验失败：删除临时文件，保留旧快照
→ 空购物车：立即原子启用
→ 非空购物车：标记“新数据待启用”
→ 当前订单完成或取消：启用待切换快照
```

- 每个购物车创建时绑定一个商品/促销快照版本。
- 非空购物车不得在后台同步后突然变价。
- 新快照下载失败不影响当前购物车。
- 多部手机各自维护检查时间、下载状态和当前快照，互不影响。

### 10.3 离线行为

- 始终允许使用最后一份有效快照离线收银。
- 显示最后同步时间和数据过期提示。
- 单次断网、SQL 繁忙或一次同步失败只显示非阻塞状态。
- 连续失败或长期未更新显示明显警告。
- 没有任何有效电脑快照时，不能使用电脑商品自动收银；手机本地商品仍可使用。

## 11. 促销验证与计算规则

### 11.1 验证状态

每条或每类促销必须处于以下状态之一：

```text
VERIFIED      已通过鸣盛黑盒测试，可自动计算
UNVERIFIED    数据已同步，但计算语义尚未确认
INVALID       数据关系缺失、参数异常或无法安全解析
UNKNOWN_TYPE  出现未支持的新规则类型
```

### 11.2 未验证或异常促销的收银行为

- 商品可以查询和加入购物车。
- 自动结账前明确提示该商品促销尚未验证或数据异常。
- 自动价格不能静默退回普通价。
- 收银员明确选择手动价格后可以继续结账。
- 手动价格优先于自动促销，仅作用于当前购物车商品行，不写回电脑端。

### 11.3 时间和促销计算

- 电脑端同步完整日期、星期和时段规则。
- 手机端使用 `America/Argentina/Buenos_Aires` 本地判断是否生效。
- 离线时继续依据最后快照和阿根廷时间计算。
- 促销优先级、是否叠加、阶梯选取、组合商品分摊和金额舍入全部以鸣盛黑盒结果为准。
- 不允许根据活动名称、字段名或常见零售经验猜测规则。

### 11.4 小数数量

- 手动输入的非整数数量不参与任何自动促销。
- 非整数数量商品行只计算正常单价或手动价格。
- 整数数量仍需满足实际商品映射、时段和已验证规则才应用促销。

### 16.2 失败策略

- SQL 连接失败：记录原因，保留旧快照。
- 数据库繁忙：非错误状态，等待下一周期或手动队列。
- 两次读取不一致：丢弃本轮，保留旧快照。
- 促销关系异常：写入验证问题；若影响安全发布则拒绝整份快照，否则将受影响规则标记为 `INVALID`。
- SQLite 生成或自检失败：删除临时文件，保留旧快照。
- 手机下载或哈希失败：删除临时文件，继续旧快照。
- 手机不支持 schema：拒绝启用并提示升级。

### 16.3 日志保护

- 日志可记录阶段、耗时、表行数、快照编号、失败代码和来源 IP。
- 不记录完整商品数据。
- 不记录完整 Token、带 Token 的 URL 或 SQL 连接秘密。
- SQL 查询来自固定白名单；可记录查询标识，不记录可泄漏敏感信息的动态文本。

## 17. 促销黑盒验证方案

每一种当前启用促销都必须在鸣盛电脑端创建或选择可复现案例，不需要完成最终付款。

每类至少验证：

1. 门槛前一档数量。
2. 正好达到门槛。
3. 超过门槛。
4. 多阶梯之间的边界。
5. 活动开始前、开始时、结束前、结束后。
6. 有效星期和无效星期。
7. 活动商品与非活动商品。
8. 同组商品不同组合。
9. 简单促销和复杂促销同时存在时的优先级。
10. 两个复杂促销同时命中时是否叠加。
11. 带 `2099.99` 等小数价格的商品。
12. 最终行金额和订单总额的小数处理。

测试记录至少包含：

```text
促销类型和源主键
测试时间（ART）
商品 GID / 条码
各商品数量
鸣盛显示的单价、优惠、行小计和订单总额
是否需要会员或价格类型
手机端期望结果
结论：已验证/仍不明确
```

只有证据充分的规则才能从 `UNVERIFIED` 改为 `VERIFIED`。

## 18. 自动化测试与安全验证

### 18.1 电脑端

至少覆盖：

- 查询白名单拒绝任何非 `SELECT` 语句和未知表。
- Windows 集成身份验证配置不保存密码。
- 繁忙和安静窗口状态转换。
- 双读相同发布、双读不同拒绝。
- 超时和断线保留旧快照。
- 商品、分类、单位、促销规范化。
- 已知异常映射进入 `validation_issues`。
- SQLite v2 完整性和 manifest 计数。
- 原子发布、历史保留和并发发布锁。
- `/v2/manifest.json`、`/v2/latest.db`、Token 和旧端点回归。

### 18.2 手机端

至少覆盖：

- v2 schema 支持和拒绝不兼容版本。
- SHA-256、大小、结构和计数校验。
- 空购物车立即切换，非空购物车延迟切换。
- 同步后当前购物车价格不变。
- 停用商品搜索和扫码行为。
- 同步商品只读、本地商品仍可编辑。
- 前台周期、恢复前台、进入收银和新订单触发。
- 离线使用最后快照。
- 非整数数量排除全部促销。
- 未验证/异常促销阻止自动结账，手动定价后可继续。
- 每个已验证促销类型的边界和阿根廷时间测试。
- 精确十进制价格、折扣参数和最终金额处理。

### 18.3 真实收银电脑安全验证

- 同步期间正常执行鸣盛扫码、开单、改价和查询。
- 监测 SQL 等待和阻塞，确认同步查询不阻塞业务事务。
- 验证同步前后鸣盛表记录、MDF/LDF、配置和 SQL 服务状态未被程序修改。
- 检查程序实际执行的查询均来自只读白名单。
- 数据库持续繁忙时，确认同步主动延迟而不是强行读取。

## 19. 推荐实施顺序

1. 固化 SQL 查询白名单、只读连接和模拟数据测试，不接触真实主库写操作。
2. 完成变化探测、安静窗口、双读规范化和安全失败路径。
3. 完成 SQLite v2、manifest v2 和兼容 HTTP 端点。
4. 完成手机 v2 下载、校验、快照仓库和订单边界切换。
5. 完成商品、分类、单位、停用状态和精确十进制价格接入。
6. 逐类完成促销黑盒测试，再逐类启用手机自动计算。
7. 完成两端周期设置、状态 UI、警告和手动同步队列。
8. 完成真实收银电脑的低干扰验收。
9. 运行完整回归，重建电脑端 EXE/ZIP 和 Android APK，再进行端到端联调。

不得在促销语义尚未验证时一次性宣称所有促销已支持。

## 20. 多角色开发边界

### 20.1 电脑端后端

负责 SQL 只读适配、安全门、规范化、SQLite v2、manifest、HTTP 和测试。不得修改 PySide6 页面布局、Android 代码或鸣盛数据。

### 20.2 电脑端前端

负责数据源、周期、同步状态、立即同步和取消等待界面。不得执行 SQL、写快照或修改防火墙。

### 20.3 手机端后端

负责 v2 协议、快照、精确十进制金额/数量、促销模型和定价。不得修改电脑端协议实现或连接 SQL Server。

### 20.4 手机端前端

负责同步设置、状态、只读商品提示、小数数量和促销安全提示。不得自己解析数据库或计算促销。

### 20.5 促销验证与验收

负责黑盒证据、自动化测试、构建、真实电脑联调和安全审查；不根据活动名称猜测业务公式。

所有角色必须保持模块化和函数拆分，不进行与本轮无关的重构。

## 21. 不做范围

本轮不做：

- 双向商品同步。
- 手机写回 `MS2011`。
- 手机销售流水同步到电脑。
- 库存、会员、供应商、采购和财务同步。
- 公网同步或内网穿透。
- SQL Server 服务维护。
- MDF/LDF 文件复制。
- 对鸣盛程序进行反编译、注入或修改。
- 未经黑盒验证就猜测促销规则。
- Android 后台常驻同步；本轮仅 App 前台自动检测。
- 多手机设备管理和设备间销售合并。

## 22. 完成标准

只有同时满足以下条件才能宣布完成：

- 电脑端只执行固定白名单 `SELECT`，没有修改鸣盛软件、数据库或 SQL 服务。
- 同步查询在真实收银操作中不造成阻塞或可感知卡顿。
- 数据库繁忙时同步会延迟、取消或重试，不会强制执行。
- 商品、分类、单位、停用状态和已经应用的复杂促销候选进入规范化 v2 快照。
- 金额和折扣参数保持原始十进制数值，`2099.99` 不改变为其他单位，也不使用二进制浮点参与结账。
- 默认配置且数据库空闲、网络正常时，新修改通常在 60 秒内供手机新购物车使用。
- 手机离线时继续使用最后有效快照。
- 非空购物车不会因同步中途变价。
- 非整数数量不参与促销。
- 每种当前启用促销类型均完成鸣盛黑盒验证，手机结果与鸣盛一致。
- 未验证、异常和未知促销不会被静默忽略。
- `AGT_MAIN` 兼容模式、手机本地商品编辑、手动改价、手动导入、CSV 导入、销售和日账无回归。
- 电脑端自动化测试、Android 测试、真实电脑安全验证和端到端联调全部通过。
- 新 EXE/ZIP 和 APK 在代码验收后重新生成并验证版本、时间戳和 SHA-256。

## 23. 实施前仍需取得的证据

以下不是未决产品方向，而是进入对应促销实现前必须取得的证据：

- 各 `CXTSALEKOU` 原始数值的实际折扣含义。
- 简单促销与复杂促销同时命中时的优先级。
- 多个复杂促销同时命中时是否叠加。
- 固定总价超过门槛后的重复分组或余数计算方式。
- 混合商品固定总价的组合选择和金额分摊方式。
- 开始/结束日期、星期和时段边界是否包含端点。
- 鸣盛对十进制中间结果和最终金额的舍入时点。

取得证据前，对应规则保持 `UNVERIFIED`，不得自动结账。
